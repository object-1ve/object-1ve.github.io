../和./和/

  1. ../：
    • 这表示“上一级目录”。当你在一个目录结构中移动时，../ 用于回到当前目录的父目录。

  2. ./：
    • 这实际上代表“当前目录”。尽管在很多情况下可以省略，./ 明确指出你正在引用当前目录下的资源或文件。

    • 这个斜杠在路径中用于分隔不同的目录或文件名。
    • 当作为路径的第一个字符出现时，/ 表示根目录。在Web服务器的上下文中，这通常意味着网站的主目录。

1. 布尔代数、逻辑代数

布尔代数是用于逻辑运算的一种代数结构，主要包含以下元素和运算：

  • 0 和 1: 代表布尔值，0 通常表示假，1 通常表示真。
  • 正逻辑: 1 表示真，0 表示假。
  • 负逻辑: 1 表示假，0 表示真。

2. 进制和码制转换

常见进制和码制包括二进制、八进制、十六进制、十进制以及 8421BCD 码。各进制之间的转换关系如下：

  • 二进制（Binary）: 基数为 2，用 0 和 1 表示。
  • 八进制（Octal）: 基数为 8，用 0-7 表示。每个八进制数对应 3 位二进制数。
  • 十六进制（Hexadecimal）: 基数为 16，用 0-9 和 A-F 表示。每个十六进制数对应 4 位二进制数。
  • 十进制（Decimal）: 基数为 10，用 0-9 表示。
  • 8421BCD 码: 每个十进制数用 4 位二进制数表示。

转换示例：

  • 二进制到十六进制: 二进制数 1101 1010 转换为十六进制数为 DA。
  • 十六进制到二进制: 十六进制数 B 转换为二进制数为 1011。
  • 8421BCD 码: 十进制数 19 转换为 8421BCD 码为0001 1001。

3. 基本逻辑关系

基本逻辑运算包括与（AND）、或（OR）、非（NOT），扩展逻辑门包括与非门（NAND）、或非门（NOR）、异或门（XOR）、同或门（XNOR）。

  • 与（AND）: 两个输入都为 1 时输出为 1，否则输出为 0。
  • 或（OR）: 只要有一个输入为 1，输出为 1。
  • 非（NOT）: 输入为 0 输出为 1，输入为 1 输出为 0。
  • 与非门（NAND）: 与运算后取反。
  • 或非门（NOR）: 或运算后取反。
  • 异或门（XOR）: 输入相同时输出为 0，输入不同时输出为 1。
  • 同或门（XNOR）: 输入相同时输出为 1，输入不同时输出为 0。

4. 真值表、逻辑表达式、最小项和标准与或式

  • 真值表: 显示逻辑运算中所有输入组合的输出结果。
  • 逻辑表达式: 使用逻辑运算符（如 AND、OR、NOT）表示逻辑关系。
  • 最小项: 对于 n 个变量，每一个变量的组合（如 00、01、10、11）称为最小项。
  • 标准与或式: 用最小项组合成的逻辑表达式。
  • 最简式: 通过化简逻辑表达式得到的简洁形式。

5. 公式定理规则

  • 代入规则: 根据已有的逻辑表达式进行代入计算。
  • 反演规则: 使用德摩根定律进行反演操作。

公式化简法

公式化简通常使用以下方法：

  1. 代入规则: 替换表达式中的变量。
  2. 结合律和分配律: 进行组合和分配以简化表达式。
  3. 反演规则: 使用德摩根定律等进行反演简化。
  4. Karnaugh 图法: 通过 Karnaugh 图（卡诺图）直观地进行化简。

6. 卡诺图

卡诺图（Karnaugh Map，K-Map）是一种用于化简布尔代数表达式的工具。通过图形化的方法，可以轻松识别和消除冗余项。

  • 逻辑相邻项: 在卡诺图中，逻辑相邻项指的是仅有一个变量不同的相邻格子。例如，对于两个变量 A 和 B，它们的四个组合（00、01、11、10）在卡诺图中排列成一个 2x2 的矩阵，任意相邻格子代表逻辑相邻项。
  • 几何相邻: 直接相邻或通过边界相邻的格子。例如，边界相邻的格子 00 和 10 可以视为相邻。

卡诺图化简法

  • 圈越大越好: 尽可能圈住多个“1”来消除更多的变量。
  • 圈的个数越少越好: 用最少的圈覆盖所有的“1”。
  • 每个圈内有不同的“1”: 每个圈应该尽可能覆盖不同的“1”。

13. 触发器

触发器是具有记忆功能的基本单元，用于构建时序逻辑电路。常见触发器有：

  • RS 触发器: 基于与非门（NAND）或或非门（NOR）的简单触发器。输入为 R 和 S，具有置位和复位功能。
  • D 触发器: 数据触发器，输入为 D 和时钟信号，D 输入直接决定输出状态。
  • JK 触发器: 改进型 RS 触发器，输入为 J 和 K，解决了 RS 触发器的不确定状态。
  • T 触发器: 翻转触发器，输入为 T，T=1 时翻转输出状态，T=0 时保持当前状态。

触发器的特性方程及转换

  • D 触发器: ( Q_{next} = D )

  • T 触发器: ( Q_{next} = T \oplus Q )

3. 弱实体（Weak Entity）

弱实体是依赖于另一个实体（称为“强实体”）的存在而存在的实体。弱实体没有独立的主键，通常依赖于强实体的主键，并使用一个部分键（Partial Key）来唯一标识。例如，订单和订单项，订单项作为弱实体依赖于订单的存在。

4. 强实体（Strong Entity）

强实体是独立存在的实体，具有独立的主键来唯一标识每个实体实例。例如，学生实体中的每个学生都有一个唯一的学生ID作为主键。

5. 部分键（Partial Key）

部分键是用于唯一标识弱实体的属性，但不能单独作为主键。例如，在订单项（弱实体）中，订单ID和项ID组合在一起可以唯一标识一个订单项。

6. 参与约束（Constraints on Participation）

参与约束描述了实体在关系中的参与程度，可以是完全参与（Full Participation）或部分参与（Partial Participation）。

  • 完全参与（Full Participation）：每个实体实例必须参与关系。例如，每个员工都必须属于某个部门。
  • 部分参与（Partial Participation）：实体实例可以不参与关系。例如，有些学生可能没有选课。

事务概念：ACID特性含义、举例

事务（Transaction）

事务是一个或多个操作的集合，这些操作作为一个单一的逻辑单元执行。在数据库中，事务保证了数据的完整性和一致性。事务具有以下ACID特性：

ACID特性

  1. 原子性（Atomicity）

    • 含义：事务中的所有操作要么全部完成，要么全部不完成。事务的任何部分的失败都会导致整个事务的失败，并且数据库会回滚到事务开始前的状态。
    • 举例：银行转账操作，假设从账户A转账100元到账户B。这个事务包含两个操作：从账户A中扣除100元，以及在账户B中增加100元。原子性保证了要么两个操作都成功，要么都失败。如果在扣除账户A的100元后发生错误，则事务回滚，账户A的100元会恢复。
    • 银行转账,a给b100元,包含了,a-100,b+100,原子性保证了,若a-100发生错误,则事务回滚,a+100

  2. 一致性（Consistency）

    • 含义：事务开始前和结束后，数据库必须处于一致的状态。事务完成后，所有数据的状态必须符合数据库的所有约束条件。
    • 举例：在银行转账操作中，假设账户A和账户B的初始总余额是1000元。一致性保证在事务完成后，无论转账发生了什么，总余额仍然是1000元。

  3. 隔离性（Isolation）

    • 含义：多个并发事务互不干扰，事务的执行效果不受其他事务的影响。事务的中间状态对其他事务不可见。
    • 举例：在并发执行的情况下，如果事务T1正在执行A转账给B操作，而事务T2也想访问账户A或B，隔离性保证了T2不会看到部分完成的转账操作，从而避免了数据不一致的问题。

  4. 持久性（Durability）

    • 含义：一旦事务完成（提交），其对数据库的所有更改都将被永久保存，即使系统故障也不会丢失。
    • 举例：在银行转账操作中，一旦事务提交，从账户A扣除100元并增加到账户B的操作是持久的，即使之后系统崩溃或断电，这些更改也不会丢失。

举例说明

举例：银行转账操作

假设我们有一个银行系统，其中包含以下两个账户：

  • 账户A：余额500元
  • 账户B：余额300元

我们进行一次转账操作，从账户A转账100元到账户B。这一操作可以表示为一个事务，具有以下步骤：

  1. 从账户A扣除100元。
  2. 向账户B增加100元。

ACID特性在此事务中的应用

  1. 原子性

    • 如果在执行过程中，扣除账户A的100元成功，但增加账户B的100元失败，系统会回滚事务，账户A的100元会恢复。确保整个转账操作要么完全成功，要么完全失败。

  2. 一致性

    • 转账前：账户A余额500元，账户B余额300元，总余额800元。
    • 转账后：账户A余额400元，账户B余额400元，总余额800元。事务前后，总余额不变，保持一致性。

  3. 隔离性

    • 在转账操作进行时，其他任何读取账户A或账户B余额的事务，都不会看到中间状态（例如，账户A余额已扣除100元，但账户B余额尚未增加100元）。只有在转账事务提交后，其他事务才会看到账户A和账户B的最终状态。

  4. 持久性

    • 一旦转账操作成功并提交，即使系统随后崩溃或断电，账户A的余额变为400元，账户B的余额变为400元的结果也将被永久保存，不会丢失。

通过上述示例，可以清晰地理解ACID特性在事务中的具体应用和重要性。

关系数据库的重要概念及示例

1. 候选键（Candidate Key）

  • 含义：候选键是一个表中的一个或多个字段的组合，这些字段可以唯一地标识表中的每一行。一个表可以有多个候选键。
  • 例子：在一个学生表（Student）中，学生ID（StudentID）和社会安全号码（SSN）都可以作为候选键，因为它们都能唯一标识一个学生。

2. 主键（Primary Key）

  • 含义：主键是从候选键中选出的一个键，用于唯一标识表中的每一行。一个表只能有一个主键，且主键的值不能为NULL。
  • 例子：在学生表（Student）中，StudentID可以被选为主键，因为它能唯一标识每个学生。

3. 外键（Foreign Key）

  • 含义：外键是一个表中的一个或多个字段，它们引用另一个表的主键，用于建立两个表之间的关系。外键约束确保引用的完整性。
  • 例子：在课程注册表（Enrollment）中，StudentID可以作为外键，引用学生表（Student）中的主键StudentID，以表示学生与所选课程之间的关系。

4. 关系（Relation）

  • 含义：关系是关系数据库中的一个表，由行和列组成，每一行代表一个元组，每一列代表一个属性。
  • 例子：学生表（Student）就是一个关系，其中每一行代表一个学生，每一列代表学生的一个属性（如姓名、年龄）。

5. 属性（Attribute）

  • 含义：属性是表中的列，表示实体的某个特征或特性。
  • 例子：在学生表（Student）中，StudentID、Name、Age是表的属性。

6. 元组（Tuple）

  • 含义：元组是表中的一行，表示实体的一个实例。
  • 例子：在学生表（Student）中，一行数据如 (1, ‘Alice’, 20) 是一个元组，表示一个具体的学生。

7. 约束（Constraints）

  • 含义：约束是对表中数据的规则，用于确保数据的完整性和一致性。
    • Not Null：确保列的值不能为空。
    • Check：确保列的值满足指定的条件。
    • Unique：确保列的值唯一，不重复。
    • Datatype：指定列的数据类型，如整数、字符串、日期等。

简答题

1. 数据库、数据库系统与数据库管理系统的区别和联系

数据库（Database）

  • 定义：数据库是一个有组织的数据集合，存储在计算机系统中，能够高效地存储、检索和管理数据。
  • 例子：一个公司的人事数据库，包括员工的信息（姓名、年龄、职位等）。

数据库管理系统（DBMS，Database Management System）

  • 定义：数据库管理系统是用于创建、管理、操作和维护数据库的软件。它提供了与数据库交互的接口。
  • 例子：Oracle、MySQL、Microsoft SQL Server等。

数据库系统（DBS，Database System）

  • 定义：数据库系统是指包含数据库、数据库管理系统及相关应用程序的整体。
  • 例子：一个公司的HR系统，包含员工数据库（数据）、用于操作该数据库的MySQL（DBMS），以及HR管理软件（应用程序）。

联系

  • 数据库是数据库系统的核心部分，存储实际的数据。
  • 数据库管理系统是管理和操作数据库的工具。
  • 数据库系统是综合的体系，包括数据库、DBMS以及相关的应用程序和硬件支持。

2. 数据和程序的物理独立性和逻辑独立性

物理独立性（Physical Independence）

  • 定义：指用户和应用程序不需要了解数据的物理存储结构，即使数据的存储结构发生变化，也不影响应用程序。
  • 例子：如果数据库管理员改变了数据存储的文件系统或存储设备，应用程序依旧可以正常运行。

逻辑独立性（Logical Independence）

  • 定义：指用户和应用程序不需要了解数据的逻辑结构变化，即使数据的逻辑结构（如表的字段、视图等）发生变化，也不影响应用程序。
  • 例子：如果数据库管理员增加一个新字段，现有的应用程序依旧可以使用，不受影响。

3. 数据库故障的种类及恢复步骤

数据库故障种类

  1. 事务故障：事务在执行过程中因错误或异常中断。
  2. 系统故障：系统硬件或软件故障导致数据库服务中断。
  3. 介质故障：存储介质（如磁盘）损坏导致数据丢失。

恢复步骤

  1. 事务故障恢复
    • 步骤：回滚未完成的事务，恢复到事务开始前的状态。
    • 工具：事务日志、回滚操作。
  2. 系统故障恢复
    • 步骤：重启系统，利用日志文件恢复未提交事务。
    • 工具：检查点（Checkpoint）、日志文件。
  3. 介质故障恢复
    • 步骤：从备份文件中恢复数据库，然后利用日志文件恢复最近的事务。
    • 工具：备份文件、日志文件。

4. 事务的4个性质及其含义和必要性

  1. 原子性（Atomicity）

    • 含义：事务中的所有操作要么全部成功，要么全部失败。
    • 必要性：确保事务在错误情况下可以回滚，保持数据一致性。

  2. 一致性（Consistency）

    • 含义：事务在执行前后，数据库必须保持一致的状态。
    • 必要性：保证事务完成后，数据库的完整性约束和业务规则不被破坏。

  3. 隔离性（Isolation）

    • 含义：多个并发事务互不干扰，中间状态对其他事务不可见。
    • 必要性：确保事务在并发执行时不互相影响，避免脏读、幻读等问题。

  4. 持久性（Durability）

    • 含义：事务完成后，其对数据库的更改是永久的，即使系统崩溃也不会丢失。
    • 必要性：确保提交的事务不会丢失，保证数据的持久性。

5. 登记日志文件的两条原则

  1. 日志先行（Write-Ahead Logging）

    • 原则：在修改数据库之前，先将相关日志记录写入日志文件。
    • 必要性：确保即使在故障发生时，也可以根据日志记录进行恢复。

  2. 立即写入（Immediate Logging）

    • 原则：一旦事务操作完成，立即将日志记录写入稳定存储（如磁盘）。
    • 必要性：确保日志记录及时持久化，防止系统崩溃后数据丢失。

6. 数据库完整性与安全性区别

完整性（Integrity）

  • 定义：完整性是指数据库中数据的准确性和一致性，包括实体完整性、参照完整性和域完整性。
  • 保障方式：通过约束（如主键约束、外键约束、检查约束等）来保证数据的完整性。

安全性（Security）

  • 定义：安全性是指保护数据库免受非法访问和恶意破坏，确保数据的保密性和完整性。
  • 保障方式：通过用户认证、权限控制、加密等手段来保护数据的安全性。

7. 一级、二级、三级封锁协议及解决的数据不一致性问题

一级封锁协议（Level 1: Read Uncommitted）

  • 定义：事务可以读取未提交的数据。
  • 问题：解决脏读问题，但仍存在不可重复读和幻读问题。

二级封锁协议（Level 2: Read Committed）

  • 定义：事务只能读取已提交的数据。
  • 问题：解决脏读问题，但仍存在不可重复读和幻读问题。

三级封锁协议（Level 3: Repeatable Read）

  • 定义：事务在读取数据时，会锁住所读取的数据，直到事务结束。
  • 问题：解决脏读和不可重复读问题，但仍可能存在幻读问题。

序列化级别（Serializable）

  • 定义：事务完全隔离，事务在执行时不会看到其他事务的中间结果。
  • 问题：解决脏读、不可重复读和幻读问题，确保数据一致性。

关系规范化判断：1NF、2NF、3NF、BCNF

第一范式（1NF）

定义：所有属性都是原子的，即属性值不可再分。 要求：

  • 每个列的值都是不可分割的原子值。
  • 每个列中只包含一种类型的数据。

例子：一个包含姓名和电话号码的表，若每个人只能有一个电话号码，该表就是第一范式。如果一个人有多个电话号码，需要拆分成多行来满足第一范式。

第二范式（2NF）

定义：在第一范式的基础上，每个非主属性完全依赖于主键。 要求：

  • 满足第一范式。
  • 每个非主属性必须完全依赖于主键，而不是部分依赖。

例子：在一个学生课程表中，如果有学生ID、课程ID和成绩三个属性，并且主键是（学生ID，课程ID），那么成绩应该完全依赖于（学生ID，课程ID），而不是依赖于其中一个部分。

第三范式（3NF）

定义：在第二范式的基础上，每个非主属性不依赖于其他非主属性。 要求：

  • 满足第二范式。
  • 每个非主属性只依赖于候选键，而不传递依赖于候选键。

例子：在一个员工表中，如果有员工ID、部门ID和部门名称三个属性，并且主键是员工ID，那么部门名称应该直接依赖于部门ID而不是通过员工ID间接依赖。

巴斯-科德范式（BCNF）

定义：在第三范式的基础上，任何一个非平凡的函数依赖关系的左边都应该是候选键。 要求：

  • 满足第三范式。
  • 对于任意的函数依赖 X -> Y，X 应该是一个候选键。

例子：在一个课程安排表中，如果有课程ID、教师ID和上课时间三个属性，并且一个教师只能教授一个课程，那么教师ID应该是候选键。

关系规范化 (Relational Normalization)

关系规范化是将关系模式转换为一系列满足特定范式要求的模式的过程，以减少数据冗余和异常。

  • 第一范式 (1NF): 要求关系中的每个属性都是原子性的，不能有重复的组。
  • 第二范式 (2NF): 在1NF的基础上，要求所有非主属性完全依赖于候选键。
  • 第三范式 (3NF): 在2NF的基础上，要求非主属性不依赖于其他非主属性。
  • Boyce-Codd范式 (BCNF): 在3NF的基础上，要求每个函数依赖的左边都是候选键。

关系的分解 (Decomposition of Relations)

关系的分解是将一个复杂关系分解为多个简单关系的过程，以提高数据库的规范性。

  • 无损连接分解 (Lossless Join Decomposition): 确保分解后的关系可以通过自然连接恢复到原始关系。
  • 保持函数依赖分解 (Preservation of Functional Dependencies Decomposition): 确保分解后的关系仍然满足原有的函数依赖。

并发控制

必要性

并发控制是为了保证在多个事务同时访问数据库时，数据的一致性和完整性。由于多个用户可能同时对数据库进行读写操作，如果不进行并发控制，可能会导致数据不一致、丢失或破坏。

并发操作可能引起的问题

  1. 丢失更新：两个事务同时更新同一数据，导致一个事务的更新被另一个事务覆盖。
  2. 不可重复读：一个事务在两次读取同一数据时，读到的数据不同，因为在两次读取期间，另一事务修改了数据。
  3. 脏读：一个事务读到了另一个未提交事务的修改数据。
  4. 幻读：一个事务在两次查询同一条件的数据集时，读到的结果集不同，因为在两次查询之间，另一事务插入或删除了数据。

如何利用封锁解决这些问题（3级封锁协议及能解决的问题种类）

三级封锁协议的定义
  1. 一级封锁协议：事务在修改数据前必须先对其加排他锁，直到事务结束。解决了丢失更新问题。
  2. 二级封锁协议：事务在读取数据前必须对其加共享锁，读取数据后可以立即释放。事务在修改数据前必须对其加排他锁，直到事务结束。解决了丢失更新和脏读问题。
  3. 三级封锁协议：事务在读取数据前必须对其加共享锁，直到事务结束。事务在修改数据前必须对其加排他锁，直到事务结束。解决了丢失更新、脏读和不可重复读问题。
封锁的联系和区别
  • 一级封锁协议只解决了丢失更新问题，而二级和三级封锁协议解决了更多的问题。
  • 二级封锁协议在读取后可以释放共享锁，而三级封锁协议则在事务结束前都保持锁，这使得三级封锁协议能够防止不可重复读和幻读问题。

并发控制举例

假设有两个事务 T1T1T1 和 T2T2T2，它们同时对同一个数据项进行操作。 T1T1T1 读取数据项A，并在某个时刻修改它，而 T2T2T2 在 T1T1T1 修改之前也读取了数据项A，并在修改后提交。

通过三级封锁协议：

  • T1T1T1 在读取数据项A时对其加共享锁，直至事务结束。
  • T2T2T2 尝试读取数据项A时，发现已经被共享锁定，必须等待直到 T1T1T1 结束。
  • T1T1T1 修改数据项A时，加排他锁，完成操作并提交。
  • T2T2T2 获取共享锁并读取最新的数据项A，避免脏读和不可重复读。

数据库故障

分类

  1. 事务故障：由于事务逻辑错误或其他原因导致的事务中断。
  2. 系统故障：由于操作系统、数据库管理系统崩溃或其他硬件问题导致的数据库中断。
  3. 介质故障：由于磁盘损坏等原因导致的数据丢失。

日志文件的作用

  • REDO日志：用于记录已提交事务的修改操作，以便在系统故障后进行重做，恢复数据。
  • UNDO日志：用于记录未提交事务的修改操作，以便在事务或系统故障后进行撤销，恢复数据到一致状态。
日志文件登记的遵守2条原则
  1. 先写日志原则（Write-Ahead Logging, WAL）：在对数据库进行任何修改之前，必须先将相应的日志记录写入磁盘。
  2. 立即写日志原则：事务的每一步操作都必须立即记录在日志中。

故障恢复步骤

  1. 事务故障：
    • 撤销（UNDO）该事务的所有操作，使数据库恢复到事务开始前的状态。
    • 重新执行（REDO）已提交事务的操作。
  2. 系统故障：
    • 检查日志文件，找出未提交的事务并撤销（UNDO）它们的操作。
    • 重做（REDO）已提交事务的操作，确保所有已提交的事务都持久化到数据库中。
  3. 介质故障：
    • 从备份中恢复数据库。
    • 使用日志文件重做（REDO）所有已提交的事务，以确保数据库恢复到最近一次提交的状态。

故障恢复举例

假设有两个事务 T1T1T1 和 T2T2T2，在系统崩溃前 T1T1T1 已提交，而 T2T2T2 尚未提交。恢复步骤：

  1. 检查日志，发现 T1T1T1 已提交， T2T2T2 未提交。
  2. 重做 T1T1T1 的操作，确保其修改应用到数据库中。
  3. 撤销 T2T2T2 的操作，恢复到事务开始前的状态。

8.2 故障分类

  • 事务故障：分为可以预期的事务故障和非预期的事务故障。可以预期的事务故障如存款透支，非预期的如数据错误或死锁等。
  • 系统故障：可能导致未完成的事务结果写入数据库，或已提交事务的更改未写回磁盘。
  • 介质故障：通常需要重新装入转储的后备副本到新的磁盘，并结合日志文件进行REDO处理。

8.3 恢复的实现技术

  • 日志文件的作用：用于事务故障恢复和系统故障恢复，记录事务的所有变更，保证数据库的可恢复性。
  • REDO（重做）：对已提交但未写入磁盘的事务操作进行重做，以恢复数据库到故障前的状态。
  • UNDO（撤销）：对未提交的事务操作进行撤销，恢复数据库到一致状态。

8.4 恢复策略

  • 事务故障恢复步骤：扫描日志，找到故障事务并撤销所有更新操作，释放资源。
  • 系统故障恢复步骤：重启系统，使用日志文件对已提交事务进行REDO，对未提交事务进行UNDO。
  • 介质故障恢复步骤：修复或更换磁盘系统，装入最近的数据库后备副本和日志副本，进行REDO处理。

8.5 具有检查点的恢复技术

  • 引入检查点机制以减少恢复时间，只考虑最近检查点前的事务。

例子

假设我们有两个表：Students 表和 Courses 表。

主键、候选键和外键的定义

Students 表

  • 主键：StudentID（唯一标识每个学生）
  • 候选键：Email（每个学生的电子邮件地址是唯一的）和 Phone（每个学生的电话号码是唯一的）

在这个表中，我们选择 StudentID 作为主键。虽然 Email 和 Phone 都是唯一的，可以作为候选键，但我们没有选择它们作为主键。

Courses 表

  • 主键：CourseID（唯一标识每门课程）

这个表中，CourseID 是唯一的并且不为空，所以它是主键。

Enrollment 表

  • 主键：EnrollmentID（唯一标识每条选课记录）
  • 外键：StudentID（引用 Students 表的 StudentID）和 CourseID（引用 Courses 表的 CourseID）

在这个表中，EnrollmentID 是主键。StudentID 和 CourseID 是外键，用于建立与 Students 和 Courses 表之间的关系。

总结

  • 主键 是表中唯一标识每条记录的键，例如 Students 表中的 StudentID。
  • 候选键 是可以唯一标识每条记录的所有键，例如 Students 表中的 Email 和 Phone。
  • 外键 是引用另一个表中的主键或候选键的键，用于建立表之间的关系，例如 Enrollment 表中的 StudentID 和 CourseID。
